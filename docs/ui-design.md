# Chat-First 여행 에이전트 — UI 디자인 러프안

> 실제 비주얼 디자인(Claude 디자인)에 들어가기 전, 화면 구성 방향을 잡는 러프 문서.
> 확정안이 아니라 방향·옵션·컴포넌트 후보를 담는다. 여기서 나온 재료를 Claude 디자인으로 구체화한다.
>
> **레이아웃: 옵션 B(채팅 + 우측 요약 패널) 확정 — 2026-07-18 목업 비교 후 결정.**

## 1. 핵심 컨셉 — "모든 것을 채팅에서"

일반 여행앱은 `검색 → 목록 → 상세 → 예약 → 결제`가 각각 별도 페이지다. 우리는 다르게 간다.

- **단일 대화 스트림**: 여행지 검색결과, 일자별 일정, 예약 후보, 결제, 확정서까지 **전부 채팅 버블 안의 리치 카드**로 보여준다.
- **페이지 전환 없음**: 대화가 곧 내비게이션. 사용자는 화면을 옮겨다니지 않고 계속 대화한다.
- **상태는 옆에 고정**: "지금 내 여행"이 어떤 상태인지(선택한 항공·숙소·합계)는 우측 요약 패널에 항상 띄워둔다. 대화의 흐름과 현재 상태를 동시에 본다.

## 2. 레이아웃 — 옵션 B 확정

| 옵션 | 형태 | 결정 |
|---|---|---|
| A. 순수 단일 컬럼 | 메신저처럼 채팅 하나만 | 컨셉 충실·모바일 강점, 단 현재 상태는 스크롤해야 확인 |
| **B. 채팅 + 우측 요약 패널** ✅ | 우측에 "내 여행" 상태 고정 | **채택** — 여행 예약은 현재 상태를 계속 봐야 하는 태스크라 패널이 UX상 유리 |
| C. 채팅 + 카드 상세 오버레이 | 카드 탭하면 전체화면 상세 | B와 병행 가능(카드 상세만 오버레이로) |

→ **B**: 대화는 chat-first로 두되, 우측에 현재 여행 상태를 항상 보여주는 요약 패널을 고정한다.

## 4. 우측 요약 패널 — "내 여행"

현재 여행 상태를 한눈에 보여주는 영역. **장바구니 + 진행상황판** 역할.

| 구획 | 내용 |
|---|---|
| 여행 기본 | 목적지 · 기간 · 인원 |
| 담은 항목 | 선택한 명소 / 항공 / 숙소 / 액티비티 |
| 합계 | 예상 총액 (선택이 바뀌면 실시간 갱신) |
| 주요 액션 | [일정 확정]·[예약 진행]·[결제] — 누르면 **채팅에 해당 카드를 여는 바로가기**. 별도 페이지·모달로 빠지지 않는다 |

**핵심 원칙 — 모든 행동은 채팅에서 완결**: 명소 담기·항공/숙소 선택·예약·결제·확정까지
**실제 진행은 전부 채팅 카드 안**에서 대화로 이뤄진다. 우측 패널의 버튼은 그 단계를 채팅으로
열어주는 바로가기일 뿐, 패널 자체가 예약/결제를 처리하지 않는다.

**대화 ↔ 패널 동기화**: 사용자가 채팅 카드에서 명소를 담거나 항공/숙소를 선택하면
우측 패널이 **즉시 갱신**된다. 대화로 만든 결정이 패널에 쌓이고, 패널이 곧 "현재까지의 여행"이다.
즉 **행동은 채팅에서, 결과 상태는 패널에서** 본다.

## 5. 메시지 타입 & 리치 컴포넌트 (채팅 영역의 핵심)

채팅 안에서 텍스트 외에 무엇을, 어떤 형태로 보여줄지가 이 디자인의 전부다.

| 타입 | 용도 | 형태 |
|---|---|---|
| `text` | 에이전트/사용자 일반 대화 | 말풍선 |
| `agent_status` | 작업 중 표시 | "🔍 여행지 찾는 중…" + 어느 에이전트인지 |
| `quick_replies` | 다음 행동 제안 | 탭 가능한 칩 버튼들 |
| `inline_form` | 날짜·인원·예산 입력 | 채팅 안 미니 폼(날짜피커/카운터) |
| `destination_carousel` | 여행지·명소 추천 | 가로 스크롤 카드 (이미지+태그+담기) |
| `flight_results` | 항공 검색결과 | **날짜별 가격 목록** + 선택한 날짜의 항공편 리스트(시간·항공사·가격) |
| `hotel_results` | 호텔 검색결과 | **지역·날짜별 숙소 목록**(가격·평점·위치). 항공 예약 날짜에 맞춰 자동 세팅 |
| `route_plan` | 멀티 목적지 동선·여정 | 방문 순서·기간·이동(수단·시간) 개요 + A안/B안 옵션 비교 |
| `itinerary` | 일자별 일정 | 타임라인 카드 (Day별 접기/펼치기) |
| `booking_summary` | 예약 요약 | 항공+숙소+합계 카드 |
| `payment` | 더미 결제 | 결제수단·금액 카드 + [결제하기] |
| `confirmation` | 예약 확정서 | 확정번호·QR(더미) 카드 |

> 위 카드에서 "담기/선택"한 결과가 **우측 요약 패널(4장)에 반영**된다.

## 6. 대화 흐름 시나리오

### 시나리오 A — 단일 목적지 + 순차 예약 ("제주 3박4일")

**① 웰컴** — 예시 프롬프트 칩. 패널: 비어있음("여행을 시작해 보세요")

**② 정보 수집 → 명소 추천** — "제주 3박4일 2명" → 패널 📍제주·👤2명, 날짜 → 📅7.25–7.28, 명소 캐러셀에서 담기 → 패널 추가

**③ 일자별 일정** — Day별 타임라인 카드. 진행 방식 칩 `[항공 먼저]` `[숙소 먼저]` `[한번에 예약]`

**④ 항공 — 날짜별 가격 목록 → 예약** (순차의 시작)
```
│(봇) 김포→제주 날짜별 최저가예요  │
│  7/24 ₩89,000  ▸7/25 ₩112,000  │  ← flight_results (날짜별 가격 비교)
│  7/26 ₩95,000 …   [7/25 선택]   │
│(봇) 7/25 항공편이에요            │
│  09:00 대한항공 ₩112,000 [예약] │  선택→예약 → 패널 ✈️추가, 날짜 확정
```

**⑤ 숙소 — 확정된 날짜·지역으로 목록 → 예약** (앞 예약이 조건이 됨)
```
│(봇) 7.25–7.28 제주 숙소예요      │  ← 항공 예약 날짜 자동 반영
│  [제주시 ▾] 지역 필터            │
│  🏨 제주신라 ₩150,000/박 ★4.6   │  ← hotel_results (지역·날짜별 목록)
│  🏨 호텔B  …            [예약]   │  선택→예약 → 패널 🏨추가, 합계 갱신
```

**⑥ 예약 요약 → 더미 결제 → 확정** — 채팅 안에서 booking_summary → payment → confirmation

> 예약은 **버튼으로도**(패널 [예약 진행]) **채팅으로도**("항공 먼저 예약해줘") 시작 가능.

### 시나리오 B — 멀티 목적지 동선 설계 ("필리핀 보홀+세부")

여러 지역을 방문할 때 방문 순서·기간·동선을 대화로 함께 설계한다(T3 킬러 기능).
```
│ 필리핀 3월중순, 보홀이랑 세부 다 │
│(봇) 두 곳 다 좋죠! 동선 2가지예요 │  ← route_plan (옵션 비교)
│ A안 세부2박→⛴️2h→보홀2박→출국   │
│ B안 보홀먼저→세부2박(공항 가까움)│
│  [A안] [B안] [보홀 더 길게]      │
│(봇) A안 기준 일정 짜볼게요 →     │  확정 후 시나리오 A의 예약 흐름으로 연결
```

## 7. 인터랙션 패턴

- **모든 행동은 채팅에서 완결** (최상위 원칙): 명소 담기·항공/숙소 선택·예약 진행·결제·확정까지 전부 채팅 카드 안에서 진행. 우측 패널은 상태를 보여주고 채팅으로 이어주는 역할만 하며, 별도 예약/결제 화면으로 빠지지 않는다.
- **카드 내 액션 버튼**: [담기]/[선택]/[예약]/[결제] 탭 → 선택이 새 사용자 메시지처럼 흐름에 반영 **+ 우측 패널 갱신**
- **패널 버튼 = 채팅 바로가기**: 패널의 [예약 진행]·[결제]를 누르면 채팅에 예약 요약/결제 카드가 뜨고, 확인·완료는 채팅에서 진행
- **대화 ↔ 패널 동기화**: 채팅에서의 결정과 패널 상태가 항상 일치
- **인라인 폼**: 날짜·인원은 채팅 안 미니 폼으로 (페이지 이동 없이)
- **스트리밍**: 에이전트 텍스트는 타이핑되듯 스트리밍
- **낙관적 UI**: 버튼 누르면 즉시 반영 + 에이전트 작업중 표시
- **스크롤**: 최신 메시지 하단 고정, 이전 내용 위로 스크롤 (패널은 스크롤과 무관하게 고정)

### 메시지 데이터 구조 (백엔드 ↔ 프론트 계약 초안)

에이전트 응답이 `type` + `payload`를 담으면 프론트는 `MessageRenderer`가 알아서 카드를 그린다.
패널 갱신용 상태는 별도 `trip_state`로 함께 내려준다.

```json
{
  "role": "assistant",
  "type": "destination_carousel",
  "text": "제주 인기 명소예요",
  "payload": {
    "items": [
      { "name": "성산일출봉", "image": "...", "tags": ["자연", "일출"], "action": "add" }
    ]
  },
  "trip_state": {
    "destination": "제주", "dates": "7.25–7.28", "travelers": 2,
    "selected": { "flight": null, "hotel": null, "spots": ["성산일출봉"] },
    "total": 0
  }
}
```

> 이 구조가 정해지면 `backend/app/api/schemas.py`의 ChatResponse를 여기에 맞춰 확장한다.

## 9. 비주얼 톤 (러프 — Claude 디자인에서 구체화)

- **무드**: 여행의 설렘 + 미니멀한 채팅. 카드·패널은 깔끔하게, 정보 과잉 없이.
- **컬러(방향만)**: 딥블루/틸 계열 베이스 + 웜한 액센트(선셋 오렌지) → 실제 팔레트는 Claude 디자인에서
- **패널**: 채팅과 살짝 다른 배경으로 구분, 하지만 튀지 않게
- **카드**: 둥근 모서리, 가벼운 그림자, 상단 이미지 썸네일
- **타이포**: 가독성 우선, 한글 잘 어울리는 산세리프. 합계 등 숫자는 tabular-nums
- **다크모드**: 고려(채팅앱 특성상 야간 사용 많음)
